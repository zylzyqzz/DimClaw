# DimClaw 现状审计报告（基于仓库真实代码）

> 审计口径：只按当前仓库代码与本地命令结果判断，不按历史文档/路线图推断。  
> 审计时间：当前工作分支（对齐 main 代码结构进行静态核查）。

## 0. 先给结论（硬结论）

1. 这是一个**可运行的本地任务执行框架**，核心链路（提交任务 → 运行状态机 → 调技能 → 校验/重试）已经落地。  
2. 但仓库里同时存在大量“入口已给出、能力未完全实装”的模块，尤其是 `ui/`、`adapters/`、部分 browser/hands 能力。  
3. README 的功能表述和“当前真实代码”存在不一致：README 仍写“当前版本 v0.1、内置 shell_command”，而代码已经有更多内置技能、Web API、插件与 Feishu sidecar 管理代码。  

---

## 1. 入口与主链路核查（`src/main.rs`）

### 已真实落地

- CLI 子命令真实存在：`submit / run / list / schedule / doctor / server`，并且 `cargo run -- --help` 可看到这些命令。  
- `run` 路径会创建 Runtime、回捞未完成任务、可选启动 scheduler、监听 Ctrl+C 安全退出。  
- `doctor` 会检查 provider 配置和认主配置。  
- `server` 会启动 Web/API 服务并触发插件初始化与自动启动。

### 仍需注意

- CLI about 文案仍是 `V0.1`，与仓库当前模块规模（插件、市场、hands 等）不匹配，版本认知会误导外部用户。

---

## 2. agents 真实完成度

### 2.1 四主智能体（Planner / Executor / Verifier / Recovery）

**结论：可用（有兜底），但“智能”强度取决于 LLM 配置。**

- `Planner`：LLM 可用时生成结构化计划；LLM 不可用时 fallback 到单步 `shell_command` 计划。  
- `Executor`：根据计划步骤生成 `executor_decision`，最终由 Runtime 调用技能执行。  
- `Verifier`：可基于 LLM 输出 verdict；无 LLM 时按 `execution_result.exit_code` 判定。  
- `Recovery`：失败后可给出 retry/fail 决策，且无 LLM 时有默认重试策略。

### 2.2 自定义智能体

- 支持读取配置构建 `CustomAgent`，在 Runtime 的 `before_xxx/after_xxx` 阶段执行，并把结果写回 payload。

### 2.3 “hands”模块

**结论：有框架 + 少量可执行动作，但很多是演示级输出。**

- `lead/researcher` 会落盘生成示例产物（csv/markdown）。
- `browser/monitor/scheduler/collector/clip` 多为状态回报、占位文本或轻量模拟。

---

## 3. providers 真实完成度

### 已落地

- `providers` 目录中真正可用于 LLM 请求的是 `OpenAiCompatibleProvider`：
  - 会调用 `/chat/completions`；
  - 有超时、取消、HTTP 重试与退避；
  - 有响应解析与 usage 回填。

### 骨架/过渡性质

- `FallbackProvider` 基于 `core::traits::provider::Provider` 轮询兜底，但这套 `core::traits` 在当前主链路里未见真实接入。  
- `OpenAiProviderAdapter::embed` 只是把字符串字节映射成 64 维浮点向量，不是实际 embedding 模型调用，属于占位实现。

## 4. skills 真实完成度

## 4.1 真实存在的内置技能（注册到 `SkillRegistry`）

`shell_command`、browser 系列、script/file 系列、http_request、process/system 系列、schedule/service、yt_dlp/ffmpeg/whisper 均已注册。

## 4.2 能力分层判断

### A. 可直接执行系统动作（已落地）

- `shell_command`：参数校验 + 超时 + 取消 + stdout/stderr/exit_code。  
- `script_execute`：将脚本拼接为 shell/cmd/powershell 命令执行。  
- `file_read/file_write/file_list/file_move/file_copy/file_delete`：文件操作技能（受 guard 控制）。  
- `http_request`：HTTP 调用（method/header/body/timeout/cancel）。  
- `process_list/process_kill/system_monitor/service_control/schedule_task`：主要通过系统命令封装。

### B. 仅基础/占位级

- `browser_click`、`browser_fill` 明确写了“占位实现”。
- `browser_screenshot` 当前是写入“screenshot placeholder ...”文本到文件，不是真实截图。
- `browser_automator` 当前只支持 `action=open`。

---

## 5. `ui/`、`feishu/`、`adapters/` 是否骨架

- `ui/mod.rs`：是骨架（仅 `init()` 注释，无实逻辑）。  
- `adapters/mod.rs`：是骨架（仅 `init()` 注释，无实逻辑）。  
- `feishu/mod.rs`：**不是纯骨架**，已实现 sidecar 进程管理（启动/停止/状态/重启监控）、日志缓冲、socket/HTTP 同步等运行时代码。

## 6. README / Cargo / 版本宣称一致性

## 6.1 明显不一致点（优先修）

1. **版本语义混乱**  
   - `Cargo.toml` 版本仍是 `0.1.0`；CLI about 也写 V0.1；README 也说“当前版本 v0.1”。  
   - 但代码中已经存在大量“超出最小 v0.1”模块（Web API、插件、marketplace、hands、多技能）。

2. **README 对技能能力描述偏旧**  
   - README 写“内置 shell_command，未来扩展更多技能”；
   - 实际 `SkillRegistry` 已注册一长串内置技能。

3. **README 的“未来计划”与现状交叉**  
   - README 把 V0.3（飞书插件化）写成未来，但代码已有 Feishu sidecar 管理与插件接口路径。

## 6.2 测试现状与对外口径

- 测试存在不稳定点：`test_custom_agents_crud` 在本地复现 `server startup timeout`。这意味着“稳定可交付”口径需要保守。

## 7. A/B/C/D（你要求的最终清单）

## A. 当前可对外宣称的功能清单（按代码事实）

1. 本地 CLI 任务流：`submit/list/run/schedule/doctor/server`。  
2. 任务状态机与四阶段执行（planning/running/verifying/retrying + success/failed/cancelled）。  
3. Runtime 可回捞未完成任务，可取消退出。  
4. Web API 与静态页面入口可启动，包含任务、配置、技能、插件、chat 等接口。  
5. OpenAI-compatible LLM Provider（含重试/超时/解析）。  
6. 一批系统执行型技能已落地（shell/script/file/http/process/service/schedule）。

## B. 不能对外宣称、只能算规划/骨架/占位

1. `ui/`、`adapters/` 目录本身（当前只有占位 `init`）。  
2. browser 精细自动化（click/fill/screenshot 目前为占位/伪截图）。  
3. hands 中大部分“智能执行”能力（当前多为示例输出或轻量模拟）。  
4. 真正多 Provider 生产级能力（当前主链路实质依赖 openai-compatible，fallback/provider-trait 体系未形成完整落地主链）。
